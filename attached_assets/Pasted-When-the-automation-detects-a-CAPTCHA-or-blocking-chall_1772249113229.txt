When the automation detects a CAPTCHA or blocking challenge, pause the script, notify the user, let them solve it manually, then resume; also improve handling of cookie banners/newsletter modals automatically.

Acceptance criteria

Detect common CAPTCHAs (reCAPTCHA, hCaptcha, Cloudflare challenge, simple image/text prompts) and treat them as pause triggers — do not attempt to bypass or auto-solve.

When a CAPTCHA is detected:

Pause the automation loop immediately.

Show an in-app notification/modal with instructions (e.g. “CAPTCHA detected — please solve it in the opened tab”).

Provide a visible “I solved it — resume” button (disabled until CAPTCHA no longer detected OR allow manual resume with verification).

Emit a structured event to the backend (WebSocket or HTTP webhook) so remote sessions / logs know the script is paused.

Resume automatically after detection clears or when user confirms; record timestamp and action in logs.

Improve popup handling (cookie banners, newsletter modals):

Use heuristics + MutationObserver to auto-dismiss or autofill per policy (mode: dismiss default; accept or autofill_subscribe optional).

Keep a small audit log of auto-actions (selector/text/timestamp) for debugging and compliance.

Safety/privacy: default to dismiss cookie banners; require explicit opt-in to auto-accept or autofill subscriptions.

Implementation notes (suggested)

Client-side:

Inject a small script early (evaluateOnNewDocument / addInitScript) that:

Observes DOM additions and scans for overlays (cookie/newsletter) and CAPTCHA markers (iframe[src*="recaptcha"], .h-captcha, data-sitekey, challenge text, or #cf-challenge).

When CAPTCHA marker found: set window.__automationPaused = true, stop the current automation step, display an overlay UI with instructions + “resume” button, and post a message to the server via WebSocket ({type: "paused", reason: "captcha", details}).

When CAPTCHA disappears (or user clicks resume & CAPTCHA not present), set window.__automationPaused = false and notify server ({type:"resumed"}).

Server-side:

If automation runs headless, maintain a websocket/session channel to the Replit UI to push pause notifications and receive resume events.

Persist pause/resume events to logs with timestamps and session id.

UX:

The in-page overlay must clearly instruct the user which tab/window to interact with and show a debug button (“Show CAPTCHA area”) that highlights the element.

If user cannot solve, provide “Abort” to safely stop and capture a screenshot/HTML for debugging.

Tests:

Test on pages with Google reCAPTCHA v2, v3 (v3 should be considered a signal — log and optionally pause), hCaptcha, Cloudflare UW challenge.

Test cookie/newsletter handling on OneTrust, Cookiebot, and at least 3 news sites with modal popups.

Validate pause/resume flow works both for local dev and when run headless with a connected browser client.

Deliverables

Client injector script (vanilla JS) with:

CAPTCHA detection and pause/resume hooks.

Popup handler (MutationObserver heuristics).

Small UI overlay for the user to resume/abort.

Server websocket webhook handler to receive pause/resume events and relay to UI.

Minimal integration guide (where to insert evaluateOnNewDocument / addInitScript in the Replit project).

End-to-end test list and sample sites to test.
